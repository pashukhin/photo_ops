## Цель

> Пользователь запускает кластеризацию своих фото и получает обратно
> детерминированное иммутабельное **дерево** пространственно-временных кластеров
> (диапазон дат, обложка, счётчики, корзина «непригодных», пометки spacelike-аномалий).
> Только бэкенд (UI — день 6, отдельная сессия).

## Концептуальная модель (из исходной модели владельца)

- **Кластеризация осмысленна только над пространством с функцией расстояния.**
  Группировка по признакам без метрики/топологии — это AND/OR-фильтр, не
  кластеризация.
- **Возможные цели (пространства):**
  - только время — группирует в эпизоды, но не защищает от «подброшенных» фото
    (внезапная синхронизация чатов и т.п.). *Отложено.*
  - конфигурационное пространство физических параметров съёмки (ISO/выдержка/
    диафрагма/объектив/фокусное) — полезно новичкам/профи, **не сейчас (возможно,
    и не потом)**.
  - **пространство-время (lat/lon + время) — делаем первым.** Лучший кандидат на
    выявление структуры.
- **Бизнес-требования к результату (независимо от типа):**
  - результат — **дерево**: узлы = кластеры, листья = ссылки на фото;
  - результат строится по набору изображений (запрос, в т.ч. «все», или выбор);
    один владелец на набор; результат и кластеры принадлежат создателю;
  - **единожды созданные результат и кластеры неизменны**; перекластеризация =
    новый сосуществующий результат, старые не исчезают;
  - удаление мягкое и **восстановимое**; удаление кластера не трогает оригиналы;
    удаление оригинала не удаляет кластер (ссылки указывают на удалённый объект);
  - публикация — из любого непустого неудалённого кластера **любого уровня**;
  - кластеризация — только по запросу; добавление фото не пересчитывает прошлые
    результаты; результат детерминирован по (вход + параметры);
  - возможная опция «перенести производные объекты на идентичные кластеры нового
    прогона» (carry-over) — *шов на потом*.
- **Заметки** (на кластерах/рёбрах) — **вне cluster-service**: отдельный концерн
  неизвестной пока необходимости → возможный будущий `notes-service`. В `cluster-db`
  нет notes-полей; гарантируем лишь стабильные UUID v7 у узлов/рёбер, чтобы будущий
  сервис ссылался через `entity_type`+`entity_id` (ТЗ §3.6).
- **Space-time специфика:**
  - в кластеризацию не попадают фото без времени **или** без координат — они идут
    в top-level узел `not_clusterable`; прогон из-за них не прерывается;
  - исходная идея владельца — «квадрат 4-интервала СТО» как расстояние; **уточнено**
    при дизайне (см. ниже): минковский интервал знаконеопределён ⇒ не метрика;
  - алгоритм иерархический; **выбор алгоритма — за пользователем** (обяз. параметр),
    дефолт HDBSCAN/OPTICS-style — *шов на потом*; в первом срезе один алгоритм.

## Дизайн (почти финальный — выработан в сессии-инциденте; не код)

1. **Кластеризация = детерминированная иерархическая кластеризация в метрическом
   пространстве-времени.** Не ML-в-обученном-смысле, не одно-пороговая сегментация.
   Алгоритмы — unsupervised и без обучения ⇒ «нет обученной модели» сохраняется;
   «объяснимо» = инспектируемое дерево с per-node merge-distance. Результат — **то
   дерево, что выдаёт алгоритм**; приведение к неглубокому виду = сегментация, отложено.
2. **Метрика — евклидова масштабированная пространство-время:**
   `d² = (c·Δt)² + d_geo²` (`d_geo` — haversine, км; `c` — км/ч, обмен время↔пространство).
   Корректная (положительно определённая) метрика, нужная иерархическим алгоритмам.
   *Отвергнуто:* минковский интервал `(c·Δt)² − |Δr|²` как расстояние — знаконеопределён
   (spacelike ⇒ отрицателен), не метрика, ломает плотностные/иерархические алгоритмы.
3. **Причинность (световой конус) — overlay-флаг, не критерий.** Соседняя по времени
   пара с `|Δr| > v_max·Δt` (spacelike) помечается `anomaly=spacelike`; намерение
   (подброс vs два фотографа) не угадываем. Реализует физическую идею владельца, не
   делая знаконеопределённую величину метрикой.
4. **Алгоритм — agglomerative (scipy), дефолт linkage `average`**, за швом сменных
   алгоритмов (HDBSCAN/OPTICS/BIRCH позже, HDBSCAN — будущий дефолт). Дендрограмма
   = дерево нативно, максимум детерминизма/объяснимости. *Trade-off:* precomputed-
   матрица O(N²) — ок для personal-scale; «вся жизнь» → масштабируемый алгоритм (тот же шов).
5. **Модель результата — иммутабельные снапшоты-деревья.** Сущности (owner =
   cluster-service, cluster-db; кросс-ссылки UUID v7 без FK): `ClusteringResult`
   (прогон), `ClusterNode` (узел дерева), `ClusterItem` (членство фото = «ссылка»).
   Членство — на узле входа фото; состав/счётчики предков — обходом. Soft-delete
   (`deleted_at`) — *операции* отложены, колонка есть. `input_fingerprint` — якорь
   детерминизма + будущего carry-over.
6. **Топология — монолингвальный Python cluster-service** (API + compute + cluster-db),
   async через RabbitMQ (`result_id == job_id`, `pending→ready|failed`), зеркально
   паттерну photo-service↔media-worker. Вычисление — Python (scipy/HDBSCAN; «не
   изобретать велосипед»; «алгоритм-демо» — руками написанные метрика/причинность —
   живёт здесь). **Go здесь НЕ используется** (его дом — usage-плоскость, сессия 012).
7. **Учёт потребления — интеграция с usage-плоскостью из 012** (отсюда зависимость
   013→012). Вычислитель само-измеряется в **сырых провайдеро-независимых единицах**
   (`byte_seconds` = интеграл памяти по времени, `cpu_seconds`, `wall_seconds` +
   доменные счётчики) и **эмитит событие потребления** в usage-service (Go) из 012;
   ценообразование/агрегация — там. Анти-vendor-lock: только сырые величины.
8. **Reverse geocoding отложен** (закрывает `photo_ops-3iy`): алгоритму геокодинг не
   нужен — место входит координатами через метрику; человеческие admin-лейблы и
   разрыв по admin-границе — отдельная задача; заголовок кластера = диапазон дат +
   грубый коорд-хинт.
9. **Детерминизм** по (вход + параметры): стабильный порядок входа (`taken_at`, затем
   `photo_id`) ⇒ воспроизводимая структура/состав; id узлов/рёбер — per-run UUID v7.
   `timestamp`: приоритет `taken_at_utc` → иначе `taken_at_local`; право на
   space-time ⇔ есть время **и** (lat,lon). *Trade-off:* смешение tz приблизительно
   (точное приведение — ТЗ §13, шов); масштаб — см. п.4.
10. **Параметры (дефолты, переопределяемы):** `c=5` км/ч, `v_max=1000` км/ч (≈ джет),
    `linkage=average`, `algorithm=agglomerative` (в срезе зафиксирован).

### Отклонения от заготовки (честно)

- **`time_only` вместо space-time (главное).** Дизайн-разделы выше берут space-time
  за первый срез; фактически сдан `time_only`, а space-time — зарегистрированный-
  позже шов за `ClusteringMethod`. Причина (ADR-0005 §2): `time_only` требует только
  время (тестовым данным не нужен GPS), всегда применим и тестируем, а device-
  сегментация даёт ту анти-инъекционную защиту, которую space-time получил бы из
  пространственного разноса.
- **Анти-инъекция = сегментация по устройству**, а не causal/spacelike. Причинностный
  (световой конус) детектор и `anomaly=spacelike` **не реализованы** — колонка-шов
  есть, в срезе всегда NULL.
- **Параметры `c=5`/`v_max=1000` и space-time-метрика не сданы**; у `time_only`
  дефолт-параметр — `linkage=average`.
- **Планка верификации сместилась соответственно:** вместо unit-тестов «haversine +
  `d²`» и «причинностный детектор на подброшенном фото» сданы тесты
  device-сегментации / episode-gap. Остальная планка выполнена: golden-дерево/
  детерминизм, раздел `not_clusterable`, component на fake-bus, идемпотентность по
  `result_id`, contract `ListPhotoSpacetime`, e2e/UI smoke.
