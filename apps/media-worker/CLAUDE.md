# media-worker

Full media-processing worker with RabbitMQ consumer loop (Task 2.5). Handles
`ProcessPhotoJob` messages, renders image variants, and publishes
`PhotoProcessingResult` messages.

Health HTTP scaffold dropped — worker liveness is signalled by the consumer
connection remaining open (no dedicated health endpoint; liveness probe in
production should check the AMQP connection or use a container health check).

## Local context

- Python 3.12 service; `pyproject.toml` declares runtime and dev deps.
- Proto stubs live in `src/photoops_proto/` (generated by `make proto`; committed). They are excluded from ruff and mypy.
- Logging / correlation: structured JSON via `logging_setup.py`; `bind_job_context()` parses `trace_id` from the job's `correlation_id` (no Python OTel SDK — that is pb6).
- Reverse-geocoding (`geocode.py`, session 022 / ADR-0007): `reverse_geocode(lat, lon) -> GeoPlace | None` — an offline nearest-city lookup (equirectangular NN) over a vendored GeoNames extract in `data/` (`cities15000.tsv.gz` + `admin1.tsv` + `countries.tsv`, CC-BY 4.0 — see `data/NOTICE`; **no scipy/numpy**). `raw_provider_data` carries the matched city's lat/lon (photo-service reads it back as `Location.lat/lon`). `handler._process` calls it after `extract_attributes` and threads the place into `encode_result` → `ImageAttributes.place`.
- `make test-media-worker` — creates/reuses `.venv`, installs `.[dev]`, runs pytest.
- `make lint-media-worker` — creates/reuses `.venv`, installs `.[dev]`, runs ruff + mypy.
- CI job `media-worker` in `.github/workflows/ci.yml` runs the same checks using `actions/setup-python@v5` (no venv needed).

## Local invariants

- Do not regenerate proto stubs from this service; `make proto` / `make proto-check` handles drift detection at the repo root.
- `geocode.py` is defensive like `exif.py`: it MUST NOT raise into the pipeline. No GPS / out-of-range / any lookup error → `None` (coordinates are preserved, processing continues, §3.4). An offline provider has no transient class, so geocoding never enters the `0od` retry path. The vendored `data/` is committed (un-ignored from the root `.gitignore` `data/` rule) and shipped in the image (`COPY apps/media-worker /app`).
- `RabbitMqBus` is not instantiated in unit tests — it requires a live broker. It is covered by the Task 4.2 integration test.
- Storage-error handling (`photo_ops-0od`): `errors.py` `classify_storage_error` splits transient (retryable — `urllib3` HTTP errors, `ServerError` 5xx, transient `S3Error` codes) from permanent (bad image, `NoSuchKey`). `MinioObjectStore.head/download/upload` re-raise a transient as `TransientProcessingError`; `JobHandler` bounded-retries it (re-raise below the cap, give up → FAILED at the cap) and `messaging/rabbitmq.py` republishes with an incremented `x-attempt` header (no callback sleep — it would block the single `BlockingConnection`). The retry arithmetic (`messaging/retry.py`, `MAX_RETRY_ATTEMPTS`) is pure + unit-covered; the pika republish and the Minio IO bodies are `# pragma: no cover` (smoke-verified).
- Broker topology (exchange/queue/DLX/DLQ) is canonical and mirrored exactly by the TypeScript adapter in the photo-service (Task 4.2). Do not change topology constants without updating both sides.
- Entry point: `python src/main.py` — calls `app.run(load())`, which blocks on `bus.start()`.
