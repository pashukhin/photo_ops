# web

## Local context

- Next.js app for authenticated upload + the rich photo gallery; runs on `WEB_PORT` (default 3000).
- Calls api-gateway via `NEXT_PUBLIC_API_BASE_URL` (default `http://localhost:3001`); all HTTP calls are in `lib/api.ts` (`listPhotos(params) → {photos, totalCount}`, `getPhoto(id)`, auth + upload).
- App shell & routing (session 014): the app has a shared shell + auth boundary. `lib/session.tsx` (`SessionProvider` + `useSession()`) is the single source of the current user — it fetches `getCurrentUser` on mount into `{ user, status }` (`'loading'|'anonymous'|'authenticated'`) and owns `login`/`signUp`/`logout`/`refresh`; a stale mount fetch cannot clobber a later mutation (guarded by a request-id ref). Routes: `/` → `redirect('/photos')`; the `(app)` route group (`app/(app)/layout.tsx`) wraps its pages in `AuthGuard` (anonymous → `router.replace('/login')`; renders children only when authenticated) then `AppShell` (`components/shell/`: top bar — brand → `/photos`, nav Photos·Clusters·Usage with `aria-current="page"` on the active link via `usePathname`, user menu + Log out — a failed logout surfaces a `role="alert"` banner); the `AuthGuard` `loading` branch renders a non-blocking `role="status"` affordance (not a blank screen), and `SessionProvider` `console.warn`s a failed `getCurrentUser` (session 015, `photo_ops-56l`); `/photos` = `components/photos/PhotosPage` (gallery + the upload action moved from the old home page), `/clusters`, `/usage` sit in the group. `/login` is OUTSIDE the group (bare `LoginScreen` in `components/auth/`; redirects an already-authenticated visitor → `/photos`). Root `layout.tsx` wraps everything in `SessionProvider`. The session-001 `.panel` CSS is retired.
- Gallery (session 011) lives in `components/gallery/`: `PhotoGallery` (container — owns the query state, fetches via `listPhotos`, polls every `GALLERY_POLL_MS` while a photo is uploading/processing, refetches on `reloadToken` change) composes `GalleryToolbar` (search/status/sort), `PhotoTable`, `GalleryPagination`, and `PhotoDetailModal` (lazy-mounted; re-fetches `getPhoto` for a fresh presigned preview, shows the error in-dialog on failure). Sort/filter/pagination are server-side. `app/page.tsx` renders `<PhotoGallery reloadToken>` and bumps the token after an upload. Poll hardening (session 015, `photo_ops-gfs`): a transient poll error no longer stops polling (tolerates `GALLERY_POLL_MAX_ERRORS` consecutive failures), a never-settling status stops after `GALLERY_POLL_MAX_TICKS`, and a stale in-flight poll response is dropped via a generation ref bumped on every main fetch (all in `types.ts`). Shared display formatters (`FALLBACK`/`fmt`/`fmtBytes`/`fmtDimensions`/`formatLocation`) live in `components/gallery/format.ts`, imported by `PhotoTable` + `PhotoDetailModal`. Session 022: `formatLocation(location)` renders the reverse-geocoded place-tag `country / region / city` (non-empty parts; `FALLBACK` when none) shown as a Location row in `PhotoDetailModal`; `PhotoAsset.location` (`lib/api.ts`) is the optional place object from the gateway.
- UI library: Tailwind (v4, CSS-first in `app/globals.css`) + shadcn/ui primitives vendored under `components/ui/` (`cn` in `lib/utils.ts`). Theme (session 015, `photo_ops-zv7`): a single coherent light theme — the shadcn tokens in `:root` are gently warmed (soft off-white `--background`, softer ink, `--radius` 0.625rem); the session-001 dark `body` override was removed (it fought the light tokens). The `.dark` token block is defined but inert (no `html.dark`); a dark-mode toggle is out of scope. **Tailwind v4 requires an `@theme inline` block in `globals.css` mapping `--color-*: hsl(var(--*))` for the shadcn utilities (`bg-background`, `bg-card`, `bg-primary`, `text-muted-foreground`, …) to be generated at all** — without it they are inert (`photo_ops-fth`: the Dialog rendered transparent over its dark overlay, buttons had no fill). A live shell-smoke assertion guards this (a primary button's background must not be transparent). There is no `tailwind.config`; the one CSS entry point is `globals.css`. `next build` does not run ESLint (`next.config.js` `eslint.ignoreDuringBuilds`); the one canonical lint is root `make lint`.
- Upload flow: browser calls `POST /photos/upload-intents` on api-gateway to get a presigned `uploadUrl`, then does a direct `PUT` to that MinIO URL (`uploadFileToPresignedUrl`), then calls `POST /photos/{id}/complete-upload` on api-gateway to finalize. `PhotosPage` (session 015, `photo_ops-ojc`) accepts **multiple** JPEGs and runs this 3-step flow per file **sequentially and resiliently** — a failed file is recorded (`name — reason`) and the batch continues; `reloadToken` bumps after each success. Status: "Uploading i of n…" → "Uploaded ok of n." (+ "Failed: …"). Single-file is just a batch of one.
- Clustering UI (session 013) lives in `components/clusters/`: `ClusterView` (container — fetches `listClusteringResults` + `listClusteringMethods`, offers a method picker + Generate which `generateClusters` then polls `getClusteringResult` every `CLUSTER_POLL_MS` until it leaves `pending` (bounded by `CLUSTER_POLL_MAX_ATTEMPTS` — a stuck-PENDING run surfaces a timeout error instead of spinning forever, session 015 `photo_ops-n7w`), and renders a chosen result's immutable tree as a nested list). Leaf `node.items` render as photo **thumbnails** (session 015, `photo_ops-hec`): ClusterView fetches the user's ready photos once (`listPhotos`, `pageSize` cap 500) into an `id → PhotoAsset` map and `TreeNodeView` shows each item as a thumbnail `<img>`, falling back to the raw id when a photo isn't resolved (unready / beyond the cap). `app/clusters/page.tsx` renders `<ClusterView />`. Clustering client calls live in `lib/api.ts`.
- Publication editor (session 018): from a ready cluster, `ClusterView`/`TreeNodeView` shows a "Create post" button on **selectable** nodes only (`kind ∈ {leaf,internal,segment}` and `photoCount>0` — not root/not_clusterable/empty) → `createPost({resultId,nodeId})` → `router.push('/posts/:id/edit')`. `components/posts/PostEditor.tsx` (route `app/(app)/posts/[id]/edit/page.tsx`, Next-15 async `params`) loads `getPost`, resolves each `PostPhoto.photoId` to a preview **variant** thumbnail via a `listPhotos({status:['ready'],pageSize:500})` id→asset map (same client-side pattern as `ClusterView`, fallback to the id; never originals — §4.4), edits title/body (plain, no markdown) + per-photo caption + order (↑/↓) + remove, and Saves via **replace-all** `updatePost` (photos sent in display order — server canonicalizes order). A load failure (404/not owned) renders a message; a Save failure shows a banner without losing edits. Publication client calls live in `lib/api.ts` (`createPost`/`getPost`/`updatePost`).
- Share + public page + My posts (session 020): pure helpers in `lib/share.ts` (`canonicalPostUrl`/`shortDescription`/`shareText`; `WEB_ORIGIN` = `process.env.NEXT_PUBLIC_WEB_ORIGIN ?? 'http://localhost:3000'` — `NEXT_PUBLIC_*` is **build-time-inlined**, so the code default is the effective value; the compose/`.env.example` entry is parity/documentation, and it is distinct from the gateway-CORS `WEB_ORIGIN`). `PostEditor`'s **published** panel shows the absolute canonical URL + **Copy link** (URL) / **Copy share text** (the `New photo story: …` template) with one shared self-reverting `aria-live` "Copied", plus Unpublish; the whole block is hidden on a draft; publish/unpublish share a `runPublishAction(fn)` envelope. The public SSR page `app/posts/[id]/page.tsx` exports `generateMetadata` (text Open Graph — `title · Photo Ops`, `description`/`og:*`/`twitter:card`; **no** `og:image`, deferred `photo_ops-278`) over a React `cache()`-shared `getPublicPost` (dedupes the metadata + page fetch per request), and is framed with a header/footer. Owner **My posts** at `app/(app)/posts/page.tsx` renders `components/posts/PostsList.tsx` (`listPosts()` → `GET /v1/posts`, owner-scoped; loading/error/empty/rows; rows link to `/posts/:id/edit` — the summary has no slug, so the public link stays in the editor); a **Posts** entry sits in the `AppShell` nav. No proto/gateway change this session.
- Tests: `vitest run` (`make test-web`) in jsdom with `@testing-library/react` (config `vitest.config.ts`, setup `vitest.setup.ts`); component behavior (render, click, modal, polling with fake timers) is exercised there. The live UI smoke (`smoke/gallery.smoke.ts`, Playwright) runs via `make smoke-ui` against a running stack — excluded from vitest, not part of `make gate`.
- Typecheck: `tsc --noEmit` (`make typecheck` runs it across all services).

## Local invariants

- Talks only to `api-gateway`, except for presigned MinIO upload URLs (direct browser-to-MinIO PUT).
- Does not hold its own database or business state; the gateway is the source of truth.
- Auth reaches the app only through `useSession()`; `SessionProvider` (`lib/session.tsx`) is the ONLY caller of the `lib/api` auth endpoints (`getCurrentUser`/`login`/`signUp`/`logout`). Components never import those directly.
